# DOC_CANONICAL_TEMPLATE.md — Output Standard

Every document generated by any builder MUST follow this structure exactly. No variations.

---

## YAML Frontmatter (Required)

```yaml
---
doc_type: project-setup|claude-md|project-instructions|system-guide|how-to|reference|feature|process|update|setup-guide|api-guide|architecture|adr|troubleshooting|config-reference|schema-reference|command-reference
domain: claude-project|ai-docs|code|reference
builder_version: "v1.1"
generated_by: [builder_name]
generated_at: YYYY-MM-DDTHH:MM:SSZ
builder_session_id: [unique_id]

audience: beginner|intermediate|advanced|all
difficulty: easy|medium|hard
time_estimate: "5-minutes|15-minutes|30-minutes|1-hour|varies"
prerequisites: [requirement_1, requirement_2]
related_docs: [doc_title_1, doc_title_2]
tags: [keyword1, keyword2, keyword3]

status: production|draft|review|deprecated
version: "1.0"
updated: YYYY-MM-DD
deprecation_date: YYYY-MM-DD (if deprecated)

quality_score: 0-100
token_count: [number]
exit_gates_passed: [1,2,3,4,5] (if all passed, ready to ship)
exit_gates_failed: [gate_numbers] (if any failed, document why in DEBT)
---
```

**Required fields:**
- doc_type, domain, builder_version, generated_by, generated_at
- audience, difficulty, time_estimate
- status, version, quality_score, exit_gates_passed

**Optional but recommended:**
- prerequisites, related_docs, tags
- updated, deprecation_date

## 1. Title + Overview (Required)

```markdown
# [Clear Action Verb + Noun]

[One sentence: what you'll accomplish]
[One sentence: why you need this / business context]

**Search keywords:** keyword1, keyword2, keyword3
**Audience:** [beginner|intermediate|advanced]
**Time to complete:** X minutes
```

**Rules:**
- Title must start with action verb (How to, Guide to, Understanding, Creating, etc.)
- Overview ALWAYS exactly 2 sentences
- Search keywords must exist
- Must restate audience from frontmatter

**Example:**
```markdown
# How to Build a Product-Market Fit GTM Strategy

Develop a go-to-market strategy targeting a validated customer segment in 4 weeks.
This ensures your product reaches customers who need it most before scaling spend.

**Search keywords:** gtm strategy, product-market fit, go-to-market, customer validation
**Audience:** intermediate
**Time to complete:** 4 weeks
```

## 2. Prerequisites (Required if any)

If NO prerequisites: Omit this section. Otherwise:

```markdown
## Prerequisites

### Level 1: Tools & Software
- **[Tool Name] [Version]** (required|optional)
  - Check: `[exact command]`
  - Install: [direct link OR command]
  - Why: [one sentence]

### Level 2: Access & Keys
- **[What Access]** (required|optional)
  - Check: [how to verify]
  - Get: [where/who to ask]
  - Why: [one sentence]

### Level 3: Knowledge
- **[Concept Name]** (optional|required)
  - Guide: [link] (time estimate)
  - Why: [one sentence]

### Level 4: Current State
- **[What Must Be Done]** (required|optional)
  - Check: [verification command]
  - Setup: [how to set up]
  - Why: [one sentence]

**Ready?** Run this:
```bash
[verification command that proves all prerequisites met]
```
[Expected result]
```

**Rules:**
- Only use Levels that apply
- Every item has: Check + Install/Get/Guide + Why
- Boolean flags (required/optional) mandatory
- Verification test at end

## 3. Content Section (Variable by doc_type)

### For Setup/Installation/How-To Guides:

```markdown
## [Section Title]

### Step 1: [Specific Action]
1. Run: `[exact command]`
   Expected: [what appears in terminal]
   
   **AI metadata:**
   - Precondition: [what must be true before this step]
   - Postcondition: [what's true after this step]
   - Time: X minutes
   - Common error: See Troubleshooting → [error section]

### Step 2: [Specific Action]
[repeat pattern]

### Step 3: [Verification/Confirmation]
[repeat pattern]
```

**Rules:**
- One action per step
- Number steps (1, 2, 3) not bullets
- Always include "Expected:" after each command
- Always include **AI metadata:** block with precondition/postcondition
- Link common errors to troubleshooting sections

### For API/Reference Docs:

```markdown
## Endpoint: [HTTP METHOD] /[path]

**Description:** [One sentence what this does]

**Authentication:** [Required|Optional] - [type]

**Parameters:**
| Name | Type | Required | Description |
|------|------|----------|-------------|
| param1 | string | Yes | What it does |
| param2 | integer | No | What it does |

**Request Example:**
```bash
curl -X GET https://api.example.com/endpoint \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"param1": "value"}'
```

**Response Example:**
```json
{
  "status": "success",
  "data": { ... }
}
```

**Response Fields:**
| Field | Type | Description |
|-------|------|-------------|
| status | string | Request status |
| data | object | Response payload |

**Error Codes:**
| Code | Meaning | Fix |
|------|---------|-----|
| 401 | Unauthorized | Check API key |
| 404 | Not found | Verify endpoint exists |
```

**Rules:**
- One endpoint per section
- Always include request + response examples
- Always include parameter table
- Always include error codes with fixes

### For Claude Project Docs (project-setup, claude-md, project-instructions, system-guide):

```markdown
## [Section Title]

[Main content: 2-4 paragraphs max]

### Key Points
- [Point 1 with specific data/metric]
- [Point 2 with specific data/metric]
- [Point 3 with specific data/metric]

### Action Items
1. [Specific action with owner + deadline]
2. [Specific action with owner + deadline]
3. [Specific action with owner + deadline]

**AI metadata:**
- Type: [strategic direction|decision|analysis|plan]
- Owner: [who drives this]
- Timeline: [when needed]
- Dependencies: [what must happen first]
```

**Rules:**
- Limit prose to 2-4 paragraphs per section (token efficiency)
- Always use bullet points for key takeaways
- Always include action items with owners
- Include metadata for AI parsing

## 4. Expected Output / Success Indicator (Required)

```markdown
## What Success Looks Like

After completing [action], you should:

| Check | Expected | Command/Evidence |
|-------|----------|-----------------|
| [What to verify] | [What should happen] | [Exact command OR describe evidence] |
| [What to verify] | [What should happen] | [Exact command OR describe evidence] |
| [What to verify] | [What should happen] | [Exact command OR describe evidence] |

**If any check fails:** Go to Troubleshooting section below.
```

**Rules:**
- Always a table (not prose)
- Each row is independently verifiable
- Include command or evidence check
- Always reference troubleshooting

---

## 5. Troubleshooting (Required if steps exist)

```markdown
## Troubleshooting

### Error: "[Exact error message from user]"
**Root cause:** [Why this happens]
**Fix:**
1. [Specific step 1]
2. [Specific step 2]
3. [Verification step]

**AI metadata:**
- Error signature: "[Exact error from logs]"
- Precondition check: [What must be true]
- Related: [Link to related doc by title]

---

### Error: "[Second error]"
[repeat pattern]
```

**Rules:**
- Section title must be exact error message
- Always explain root cause
- Fix is numbered steps
- ALWAYS include "Error signature:" for AI matching
- 3-5 errors maximum per doc (rest go to DEBT.md)

## 6. Why This Matters (Recommended)

```markdown
## Why This Approach

We chose [approach] because:
- **[Business reason]:** [Benefit]
- **[Technical reason]:** [Benefit]
- **[User reason]:** [Benefit]

We DON'T use [alternative] because:
- **[Constraint]:** [Why not]
- **[Cost]:** [Why not]
```

**Rules:**
- 3-5 points maximum
- Be honest about tradeoffs
- Show competitive consideration

## 7. What's Next (Required)

```markdown
## After Completing This

You now have [what they accomplished]. Next steps:

1. **[Related task 1]** → See: "[Related doc title]"
   - Use if: [condition]
   - Time: X minutes

2. **[Related task 2]** → See: "[Related doc title]"
   - Use if: [condition]
   - Time: X minutes

3. **[Related task 3]** → See: "[Related doc title]"
   - Use if: [condition]
   - Time: X minutes

Choose based on your needs:
- If [scenario 1]: Start with #1
- If [scenario 2]: Follow #1 then #2
- If [scenario 3]: Focus on #3
```

**Rules:**
- ALWAYS reference by doc title (not "see above")
- ALWAYS include "Use if" condition
- ALWAYS include time estimate
- Provide decision tree if multiple paths

## NEVER Include These

❌ "Best practices" (explain specific practices instead)
❌ "Learn more" links without context (use "See: [doc title]" instead)
❌ "It's important to remember" (make it a prerequisite or gotcha)
❌ Vague instructions ("Configure properly")
❌ Inline links (move to frontmatter or metadata)
❌ Wall of text (break into sections + bullets)
❌ Theory without examples
❌ Generic disclaimers

---

## Quality Requirements

### Human Pass
- [ ] Overview is exactly 2 sentences
- [ ] Every step shows expected output
- [ ] Errors organized by signature, not cause
- [ ] All links use doc titles (not URLs)
- [ ] Next steps reference other docs
- [ ] No vague language

### AI Pass
- [ ] Frontmatter complete and valid YAML
- [ ] Prerequisites use Level 1-4 structure
- [ ] Steps have **AI metadata:** blocks
- [ ] Errors have "Error signature:" field
- [ ] Success table uses commands/evidence
- [ ] Preconditions/postconditions accurate

### Token Efficiency
- [ ] Doc size within domain budget (see config/token_budgets.json)
- [ ] No unnecessary explanation (rules explain, examples show)
- [ ] Prose sections ≤ 4 paragraphs
- [ ] Every section serves clear purpose
- [ ] No redundancy between sections
